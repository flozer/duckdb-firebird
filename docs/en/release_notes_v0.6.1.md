# duckdb-firebird v0.6.1 - Release notes

Theme: **remote Firebird UX and catalog performance**. This release removes
the first-query stall seen when `ATTACH ... (TYPE firebird)` is used against
remote databases with large catalogs.

## Highlights

- **Batched catalog column loading** - `ATTACH` now loads table column metadata
  with one batched Firebird catalog query instead of one `RDB$RELATION_FIELDS`
  query per table. This keeps the same catalog contents while avoiding
  latency multiplied by table count.
- **Remote performance validated** - benchmarked with large Firebird catalogs:
  LAN first catalog query improved from `25.26s` to `1.44s` (~17.6x), and a
  high-latency WAN case improved from a projected `16-21min` to `9.49s`
  (~100-130x).
- **Safe fallback** - if the batched metadata query is not supported by a
  server or permission profile, the connector can fall back to the previous
  per-table path. Connection, credential, and systemic permission failures are
  not hidden by the fallback; the error includes the original batch failure
  context.
- **Remote connection docs** - the usage and function manuals now document the
  correct remote connection string forms:
  `firebird://USER:PASSWORD@HOST:PORT/path/to/database.fdb` and
  `database=HOST/PORT:/path/to/database.fdb;user=USER;password=PASSWORD`.

## Validation

- CI green on Windows x64 (MSVC), Linux x64, and Firebird 3/4/5 Linux matrix.
- Catalog validation confirmed matching table and column counts before and
  after the batch catalog change.
- BLOB/data fetch paths are unchanged in this release.

## Upgrade Notes

- No SQL changes are required.
- Existing `firebird_scan(...)`, `firebird_tables(...)`, and
  `ATTACH ... (TYPE firebird)` calls continue to work.
- For remote databases, prefer the documented URL or libfbclient-style forms.
  Do not use `HOST:PORT://path/to/database.fdb`.
